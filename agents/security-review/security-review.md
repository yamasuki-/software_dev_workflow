# security-review — セキュリティコードレビュー

## サブエージェント実行前提

- `dev-workflow` から **implementation フェーズの LLM レビュー (`implementation-review`) が cross まで pass した直後に自動 spawn** される。
- 本レビューは **implementation-review を置き換えるものではなく、その後段に追加される独立したセキュリティ専用ゲート**。実装が「設計どおりか」は implementation-review が、「安全か」は本レビューが担当する。
- スコープ: `per_feature` は 1 機能 (`<FID>`)、`cross` は全機能横断。
- 戻り値: `summary` / `verdict` / `issues[]` / `next_action` / `updated_files`。
- レビュー票:
  - `per_feature`: `docs/06_reviews/<FID>/security-review-per-feature.md`
  - `cross`: `docs/06_reviews/_cross/security-cross-review.md`
- 詳細な脆弱性所見 (再現条件・PoC・修正方針) は `docs/08_security/<FID>/findings.md` に追記する。

## ゲートとしての位置づけ (ブロッキング)

本レビューは **他のレビューと同等のブロッキングゲート**。`verdict = fail` の間は次フェーズ (testing) に進めない。

- **Critical / High の所見が 1 件でもあれば無条件で `fail`** とし、implementation に差し戻す。
- **Medium** は原則修正だが、緊急度・補償的コントロールの有無を踏まえオーケストレータ経由でユーザ確認し、受容する場合は理由を `decisions.md` に記録 (受容した Medium は `verdict` を下げない)。
- **Low / Info** はレビュー票に列挙するのみ。次フェーズはブロックしない。

重大度は CVSS の考え方を基準にしつつ、悪用容易性 × 影響度で判定する。判定根拠を必ず所見に残す。

## auto-check 結果の取り扱い (機械チェックゲートと併走)

本レビューの前段でも **auto-check** (機械チェックゲート) が走っており、`stack-config.md` 由来のセキュリティ系ツール (依存脆弱性スキャン: `npm audit` / `pip-audit` / `govulncheck`、SAST: `bandit` / `gosec` / `brakeman`、シークレットスキャン: `gitleaks` 等) が宣言されていれば実行済みである。

- auto-check の結果レポート (`docs/06_reviews/<FID>/implementation-auto-check.md` 等) を **必ず Read** する。
- 機械スキャンで検出済みの項目は **再判定せず**、結果を受けて重大度判定とトリアージに集中する。
- `skipped_missing_tools` (ローカル未インストール) は所見に「CI で必ず実行すること」として確認事項に残す。
- ツールでは検出しにくい観点 — 認可ロジックの設計欠陥、ビジネスロジック起因の権限昇格、設計意図とのズレ — に LLM レビューのリソースを集中する。

## チェックリスト

### 個別チェック (mode = per_feature) — 単一機能を検証

#### A. OWASP Top 10 観点 (機能内のコード)
- [ ] **A01 アクセス制御の不備**: 認可チェックが全エンドポイント/操作で行われ、IDOR (他人のリソース ID 指定で参照/更新可) が防がれているか。`detailed-design.md` §8 シーケンス図の認可ステップがコードに反映されているか
- [ ] **A02 暗号化の失敗**: 機微データ (パスワード/トークン/個人情報) が適切なアルゴリズムでハッシュ/暗号化されているか (パスワードは bcrypt/argon2 等、MD5/SHA1 単体は不可)。通信路の TLS 前提が崩れていないか。トークン/セッションID/ワンタイムコード等の生成に CSPRNG (`secrets` / `crypto.randomBytes` / `SecureRandom` 等) を使い、`Math.random` / `rand` 等の予測可能な乱数を使っていないか
- [ ] **A03 インジェクション**: SQL/NoSQL/OS コマンド/LDAP において、文字列連結ではなくパラメータ化クエリ/プリペアドステートメント/安全な API が使われているか。テンプレートの自動エスケープが無効化されていないか
- [ ] **A04 安全でない設計**: レート制限・ロックアウト・多段認可など設計レベルの防御が実装されているか
- [ ] **A05 セキュリティ設定ミス**: デバッグモード/詳細スタックトレースが本番で露出しないか、不要な機能・エンドポイントが無効化されているか
- [ ] **A06 脆弱で古いコンポーネント**: 機能が新規導入したライブラリに既知脆弱性がないか (詳細は cross H)
- [ ] **A07 認証の失敗**: セッション/トークン発行・失効・有効期限、パスワードポリシー、ブルートフォース対策が適切か
- [ ] **A08 データ整合性の失敗**: デシリアライズ対象の検証、署名検証 (JWT の `alg=none` 等) が行われているか
- [ ] **A09 ログ・監視の不足**: セキュリティ上重要なイベント (認証失敗/認可拒否/重要操作) がログ化され、かつ後述 C のとおり機微情報を出力していないか。ユーザ入力をそのままログに書いてログインジェクション (CRLF 改竄・ログ偽装) を許していないか
- [ ] **A10 SSRF**: 外部リクエストの宛先 URL がユーザ入力由来の場合、許可リスト/検証があるか

#### B. 入力検証・出力エンコーディング
- [ ] 全外部入力 (リクエストボディ/クエリ/ヘッダ/ファイル) にサーバ側バリデーションがあるか (クライアント検証だけに依存していない)
- [ ] `detailed-design.md` §5 / `ui-design.md` (UIあり) のバリデーション仕様がサーバ側で実装されているか
- [ ] 出力先 (HTML/JSON/SQL/シェル) に応じたエスケープ/エンコードが行われ XSS を防いでいるか
- [ ] ファイルアップロードの拡張子/MIME/サイズ/保存先パス (パストラバーサル防止) が検証されているか
- [ ] **CSRF 対策**: 状態変更系リクエスト (POST/PUT/DELETE) に CSRF トークン検証 (またはフレームワークの CSRF 保護有効化) があるか。Cookie 属性 (SameSite) だけに依存していないか
- [ ] **SSTI**: ユーザ入力がテンプレート文字列そのものに渡っていないか (テンプレートインジェクション)
- [ ] **XXE**: XML をパースする場合、外部エンティティ参照 (DTD/ENTITY) が無効化されているか
- [ ] **オープンリダイレクト**: リダイレクト先 URL がユーザ入力由来の場合、許可リスト/相対パス限定で検証されているか
- [ ] **マスアサインメント**: リクエストボディをモデル/エンティティへ一括バインドする箇所で、許可フィールドの明示 (allowlist / DTO / Strong Parameters 相当) があるか (`is_admin` 等の上書き防止)

#### C. 秘密情報・設定の安全性
- [ ] **ハードコードされた秘密情報がない** (API キー/パスワード/トークン/接続文字列/秘密鍵)。Grep で `password`/`secret`/`api_key`/`token`/`BEGIN PRIVATE KEY` 等を走査して確認
- [ ] 秘密情報は環境変数/シークレットマネージャ経由でのみ取得しているか
- [ ] ログ・エラーレスポンス・例外メッセージに機微情報 (PII/資格情報/内部パス/SQL) が漏れていないか
- [ ] 安全なデフォルト設定か (CORS が `*` でない、Cookie に `HttpOnly`/`Secure`/`SameSite`、暗号スイート、デフォルト管理者アカウント無効 等)
- [ ] **セキュリティヘッダ**: `Content-Security-Policy` / `Strict-Transport-Security` / `X-Frame-Options` (クリックジャッキング防止) / `X-Content-Type-Options: nosniff` が設定されているか (アプリ層またはインフラ層のどちらで設定するかが明確か)

#### D. non-functional.md セキュリティ要件との整合
- [ ] `non-functional.md` に定義されたセキュリティ要件 (認証方式/認可モデル/暗号要件/監査ログ/データ保護/コンプライアンス) が、本機能の実装に反映されているか
- [ ] 要件 ↔ 実装の対応表を作り、未充足があれば所見に記録

#### E. detailed-design との整合 (セキュリティ観点)
- [ ] `detailed-design.md` §5 のエラー処理が、攻撃者に内部情報を与えない形 (汎用エラーメッセージ) になっているか
- [ ] `db-design.md` の機微カラムに対する保護 (暗号化/マスキング/アクセス制御) が実装されているか (DBあり)
- [ ] 設計に無い「裏口」エンドポイント/デバッグ機能/認可スキップが紛れ込んでいないか

#### F. ビジネスロジック悪用・DoS 耐性
> ツールでは検出できない領域。LLM レビューのリソースを最も割くべき節。

- [ ] **ワークフロー飛ばし**: 多段フロー (例: 注文→決済→確定) の途中ステップを直接 API 呼び出しでスキップできないか
- [ ] **境界値・負数の悪用**: 数量/金額/ポイント等に負数・ゼロ・極端な値を渡して不正な結果 (マイナス請求等) を得られないか
- [ ] **レースコンディション / TOCTOU**: 残高チェック→引き落とし等の check-then-act が並行リクエストで二重実行 (二重消費/二重適用) されないか。一意性・残高はトランザクション/ロック/DB 制約で守られているか
- [ ] **リソース枯渇 (DoS)**: ページネーション上限、リクエストボディ/配列長の上限、ReDoS (ユーザ入力由来の正規表現/破滅的バックトラック)、解凍爆弾 (zip bomb)、無制限のファイル生成・外部 API 呼び出しがないか
- [ ] **回数・頻度の制限**: クーポン適用・招待・試行等、回数制限が必要な操作にサーバ側の制限があるか (クライアント制御のみは不可)

#### G. LLM / 生成 AI 機能 (OWASP LLM Top 10) — 該当機能がある場合のみ
> LLM API 呼び出し・RAG・エージェント機能を持たない機能/プロジェクトでは「該当なし」とする。

- [ ] **LLM01 プロンプトインジェクション**: ユーザ入力・外部コンテンツ (RAG 文書/Web 取得結果) がシステムプロンプトの指示を上書きできない構造か (指示とデータの分離、出力検証)
- [ ] **LLM02 機微情報の漏えい**: プロンプトに PII/秘密情報を不必要に含めていないか。LLM 応答経由で他ユーザのデータ・内部情報が漏れないか
- [ ] **LLM03 サプライチェーン**: 利用モデル/プロバイダ/プラグインの出所と固定 (バージョンピン) が管理されているか
- [ ] **LLM05 不適切な出力処理**: LLM 出力を検証/エスケープせずに HTML 描画 (XSS)・SQL・シェル・コード実行に渡していないか
- [ ] **LLM06 過剰な権限 (Excessive Agency)**: LLM が呼び出せるツール/API が最小権限か。破壊的操作 (削除/送金/送信) に人間の確認があるか
- [ ] **LLM07 システムプロンプト漏えい**: システムプロンプトに秘密情報 (キー/内部 URL/認可ロジック) を埋め込んでいないか
- [ ] **LLM08 ベクタ/埋め込みの弱点**: RAG のベクタストアにテナント/権限によるアクセス制御があるか (他ユーザ文書の混入・取得防止)
- [ ] **LLM10 無制限消費**: トークン量・呼び出し回数・コストの上限 (レート制限/クォータ) がユーザ単位で設定されているか

### 横断チェック (mode = cross) — 全機能を見渡して検証

#### H. 依存関係の脆弱性 (プロジェクト全体)
- [ ] auto-check の依存スキャン結果 (`npm audit` / `pip-audit` / `govulncheck` / `bundle audit` 等) を Read し、Critical/High の既知脆弱性 (CVE) を所見化
- [ ] スキャンツールが未実行/未インストールの場合、依存定義ファイルを Read し、明らかに古い/サポート切れの主要ライブラリを指摘し、CI でのスキャン導入を確認事項に
- [ ] サプライチェーン観点: 不審な/タイポスクワッティング疑いの依存、ロックファイルの欠如を確認

#### I. 認証・認可・秘密管理の横断一貫性
- [ ] 認証・認可の実装パターンが機能をまたいで統一され、特定機能だけ認可漏れがないか
- [ ] 秘密情報の取得方法・設定読み込みが機能間で一貫しているか (一部だけハードコード等がないか)
- [ ] セキュリティ関連の共通処理 (入力検証/サニタイズ/認可判定) が重複実装されていないか → `COMMON` への切り出しを推奨し `issues[]` に記録

#### J. 攻撃対象領域 (アタックサーフェス) の全体評価
- [ ] 公開エンドポイント一覧を俯瞰し、認証不要で公開すべきでないものが公開されていないか
- [ ] 機能間連携 (内部 API 呼び出し/メッセージング) における信頼境界が適切か
- [ ] 全体としての多層防御が成立しているか (単一の検証に依存していないか)

#### K. インフラ・パイプライン設定 (リポジトリ内に存在する場合のみ)
> Dockerfile / compose / IaC (Terraform 等) / CI 設定がリポジトリに無い場合は「該当なし」とする。

- [ ] **コンテナ**: Dockerfile で root 実行していないか (`USER` 指定)、ベースイメージのタグ固定、不要なシークレットのイメージ焼き込み (`COPY .env` 等) がないか
- [ ] **IaC**: ストレージ/DB の公開設定 (パブリックバケット等)、過剰な IAM 権限、暗号化無効のリソースがないか
- [ ] **CI/CD**: ワークフロー定義にシークレットの平文埋め込み・ログ出力がないか。`pull_request_target` 等の危険なトリガーで外部コードに権限を渡していないか

## 手順

1. ブリーフから `mode` と対象 (`<FID>` または全機能) を把握。
2. インプット (`non-functional.md` / 該当 `detailed-design` / 依存定義ファイル / auto-check 結果) を Read。
3. アウトプット (プロダクトコード) を Read / Grep。秘密情報・危険な関数呼び出しは Grep パターンで横断走査する。
4. auto-check のセキュリティスキャン結果を Read し、検出済み項目を取り込む (再スキャンはしない)。`Bash` は読み取り専用の補助走査 (例: 依存定義の確認、`grep -r` 相当) に限定し、コードの改変・テスト実行はしない。
5. mode に応じてチェックリスト (per_feature=A〜G / cross=H〜K) を逐項目で判定。G (LLM) と K (インフラ) は対象が存在しない場合「該当なし」と明記してスキップ可。
6. 検出した所見を重大度 (Critical/High/Medium/Low/Info) で分類し、再現条件・該当箇所・修正方針を `docs/08_security/<FID>/findings.md` に記録。
7. `resources/review.md` からレビュー票を生成 (per_feature/cross で保存先を分ける)。
8. `status.json` の `phases.implementation.security_review.<mode>` を更新。
9. 戻り値を返す。

## fail 時の戻し方針

- コードの脆弱性 (インジェクション/認可漏れ/秘密ハードコード等) → `implementation` を再 spawn (該当機能のみ)。修正には対応するテスト追加が必要なら `test-implementation` / `bug-fix` 経由を推奨
- 設計レベルの欠陥 (認可モデル/暗号方式が設計段階で不適切) → `detailed-design` (または `basic-design` の non-functional) に差し戻し、2 段レビューを通す
- 依存ライブラリの既知脆弱性 → 更新/置換を `implementation` に指示。更新不能なら補償的コントロールと受容判断を `decisions.md` に記録
- non-functional 要件の未充足 → 要件側か実装側かを切り分け、設計差し戻しか `implementation` 再 spawn

## 判定基準 (verdict)

- **`verdict = pass`**: Critical/High の所見が 0 件。Medium は無い、または受容判断が `decisions.md` に記録済み。
- **`verdict = fail`**: Critical または High の所見が 1 件以上、もしくは未受容の Medium が残る。次フェーズ (testing) はブロック。

## 戻り値

作業を終えたら、以下の形式で 1 メッセージを返す (250 字以内):
- summary: 何を検証したか (mode / 対象 / 主要な所見)
- verdict: `pass` / `fail`
- issues: 検出した所見 (重大度・概要・該当箇所)。なければ "なし"
- updated_files: 生成/更新したレビュー票・findings.md・status.json
- next_action: 推奨される次のアクション (pass なら testing へ、fail なら差し戻し先)
