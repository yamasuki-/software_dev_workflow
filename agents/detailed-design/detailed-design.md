# detailed-design — 詳細設計スキル

## サブエージェント実行前提

このスキルは原則 `dev-workflow` オーケストレータから **別エージェント (サブエージェント) として spawn される** ことを想定する。

重要:
- コンテキストはフレッシュ。必要な情報はブリーフとファイルから取得すること。
- スコープは原則 **1機能 (`<FID>`) ずつ**。複数機能を同時に扱わない。
- 状態は必ず `.dev-workflow/features/<FID>/status.json` に書き戻す。
- 作業終了時は以下を返す: `summary` / `updated_files` / `open_questions` / `next_action` / `blockers`。
- 重要度 high の不明点は即時 ユーザに確認 (チャットで質問)、軽微なものは `open-questions.md` に追記。

## 役割
基本設計で確定した個々の機能について、後続の実装・テスト設計が直接実行可能な詳細レベルまで設計を落とし込む。詳細設計は **「サブ機能の定義」を軸とした9章構成の単一ドキュメント** `detailed-design.md` を機能ごとに作成する。UI・DB を持つ機能のみ、任意の追加ドキュメント (`ui-design.md` / `db-design.md`) を併せて作成する。

**詳細設計ではコードを書かない。** 処理内容・I/F は文章・表・箇条書き・Mermaid 図で定義する。

## バッチ実行と横断認識

本スキルは原則 **「フェーズバッチ」** で呼ばれる。すなわち、要件に複数機能 (`F001, F002, ...`) がある場合、各機能用のサブエージェントが並行 spawn され、本スキルは1機能 (`<FID>`) を担当する。

そのため作業前に **他機能の同時並行作業を意識** すること:

1. **他機能の同フェーズ成果物が既に存在するか確認**:
   - `docs/02_detailed_design/F<他>/*.md` が (並行 spawn のため) 既に書かれている、または書かれつつある可能性がある
   - 既存ファイルを Read で参照し、命名規約・設計パターン・データ型を **揃える**
2. **共通化の機会をメモ**:
   - 自機能の設計を進めながら「これは他機能でも使えそう」と思った要素 (サブ機能・テーブル・データ型・状態) を見つけたら、`open-questions.md` に `[COMMON 候補]` プレフィックス付きで追記する
   - 後段の `detailed-design-review` が横断的に判断して `COMMON` 擬似機能への昇格を決める
3. **既存の `COMMON` 機能 (`docs/02_detailed_design/COMMON/`) があれば最優先で参照**:
   - `COMMON` で定義された型・サブ機能・状態は再定義せず、自設計から参照する形にする

## 適用範囲・粒度のルール

- **必ず1機能ずつ完結させてから次の機能に移る。** 9章のうち一部だけ書いて中断、のような状態を作らない。
- 該当しない章・ドキュメントは「該当なし」と理由を必ず明記し、空にしない。例:
  - 状態を持たない単純機能 → §9 は「該当なし (状態は単一)」
  - UIを持たないバッチ機能 → `ui-design.md` は作成しない (必要なら §5 に CLI/API 入出力例を記載)
  - 永続化のない計算機能 → `db-design.md` は作成しない
- Mermaid を使う章 (§4 関連図、§8 シーケンス、§9 状態遷移。任意の `db-design.md` の ER図も同様) は **必ず Mermaid で書く**。テキスト表だけで済ませない。
- **コードは書かない。** 擬似コードを含め、処理内容は文章・表・箇条書きで表現する。

## 手順

### Step 1 : 対象機能の特定

1. `dev-workflow` から渡された `FID`、または `project.json` の `features` から `current_phase = detailed_design` の機能を選択。
2. 複数機能が候補ならユーザに優先順位を確認 (依存関係も考慮)。
3. `.dev-workflow/features/<FID>/status.json` を Read で読み、現在のサブタスク状況を把握。途中再開なら未完了のサブタスクから着手。

### Step 2 : 基本設計の前提読み込み

以下を Read:
- `docs/01_basic_design/system-overview.md`
- `docs/01_basic_design/feature-list.md` (対象機能の行)
- `docs/01_basic_design/system-architecture.md`
- `docs/01_basic_design/non-functional.md`
- 要件定義書 (関連箇所)

### Step 3 : サブタスクをタスクリスト化

`status.json` の `phases.detailed_design.subtasks` を起点に、`TodoWrite` でタスクリストを作って進める (Cowork では `TaskCreate`)。タスクは以下に必ず細分化する:

```
[FID] 概要・トレーサビリティ (§1-2)
[FID] サブ機能定義 (§3-5: 一覧・関連図・詳細)
[FID] インタフェース定義 (§6)
[FID] シーケンス (§7-8)
[FID] 状態定義・状態遷移 (§9)
[FID] (UIあり時) UI設計 / (DBあり時) DB設計
[FID] 詳細設計レビュー (ユーザ確認)
```

各サブタスクが「重い」と感じたらさらに分割 (例: サブ機能詳細をサブ機能ごとに分ける)。

### Step 4 : 各章の作成

`resources/detailed-design.md` テンプレートをコピーして埋める。以下のポイントを必ず守る。

#### §1 概要 / §2 トレーサビリティ
- §2 の対応表は **上位ID (要件ID/仕様ID/機能ID) → サブ機能ID・章** の双方向で漏れがないようにする。
- 上位項目でカバーしないものは「対象外」と理由を明記。

#### §3 サブ機能一覧 / §4 サブ機能関連図 / §5 サブ機能詳細
- サブ機能IDは `<FID>-<連番>`。
- §4 は Mermaid `flowchart` でサブ機能間の呼び出し・データフローを描く (外部との関連も含める)。
- §5 の入出力は **型まで** 書く (string/number/Date/etc.)。
- §5 の処理内容は文章・箇条書き・表で定義する。**コードは書かない。**
- 例外/エラーケースを必ずID付きで列挙 (`E001`...)。

#### §6 インタフェース定義
- サブモジュール間 I/F を `IF-##` で列挙。**関数型 (シグネチャ表記) と入出力、処理内容のみ。コードは書かない。**
- 事前条件・事後条件・エラー時の戻り値規約を補足する。

#### §7 シーケンスパターン一覧 / §8 シーケンス図
- パターンを `SQ01, SQ02, ...` で列挙し、**正常系と代表的な異常系の両方** を含める。
- §8 は §7 の各パターンにつき Mermaid `sequenceDiagram` を1図。登場要素はサブ機能ID/I/F IDと対応させる。
- リトライ/タイムアウト/トランザクション境界などの補足を必ず書く。

#### §9 状態定義、状態遷移図、状態遷移イベント定義
- 状態が単一なら「該当なし」と明記。
- 状態ID `S0, S1, ...`、イベントID `EV01, EV02, ...` で表記統一。
- Mermaid `stateDiagram-v2` で図を必ず描く。遷移表とガード条件、不変条件 (Invariants) も書く。

#### (任意) ui-design.md
- UI を持つ機能のみ作成。画面IDは `S<連番3桁>`、項目表 (型・必須・初期値・制約)、操作・遷移、バリデーション、メッセージ、画面遷移図 (Mermaid flowchart)。

#### (任意) db-design.md
- 永続化を持つ機能のみ作成。Mermaid `erDiagram`、カラム定義表 (型・NOT NULL・PK・FK・デフォルト)、インデックス、制約、データライフサイクル。

### Step 5 : 整合性チェック

全章 (+ 任意ドキュメント) が揃ったら、以下のチェックを自分で行う。

- §2 トレーサビリティ ↔ §3/§5 サブ機能の対応の整合 (対応表の全行が実在する章・IDを指すか)
- §4 関連図の矢印 ↔ §6 I/F 定義の整合 (関連図の依存に対応する I/F があるか)
- §7/§8 シーケンスの登場要素 ↔ §3 サブ機能・§6 I/F・システムアーキテクチャ構成要素の整合
- §9 の状態・イベント ↔ §5 処理内容の整合 (遷移を起こす処理が定義されているか)
- (UIあり) UI項目 ↔ §5 入出力、(DBあり) DBカラム ↔ §5 入出力・UI項目の整合
- §5 の例外 (E###) ↔ (UIあり) UIのエラーメッセージの対応

不整合があれば該当章を修正。

### Step 6 : ユーザレビュー (チェックポイント確認)

1. 全章揃った段階で **まとめてユーザに提示** (これがチェックポイント方式の確認)。
2. `open-questions.md` の `open` 項目で本機能に関わるものを一緒に確認。
3. レビュー結果を反映。決定事項は `decisions.md` に追記。

### Step 7 : 進捗確定 (本フェーズ作業の完了)

1. `status.json` を更新:
   - `phases.detailed_design.subtasks.*` を `completed` に
   - `phases.detailed_design.status = "completed"`, `completed_at` 記入
   - **`current_phase` はまだ `test_design` に進めない** (detailed-design-review の pass を待つ)
2. `project.json` の `updated_at` を更新。
3. 戻り値で「detailed-design-review を spawn してほしい」とオーケストレータに伝える。

**重要**: 次フェーズ (`test_design`) に進めるのは **`detailed-design-review` の pass を確認した後** だけ。
