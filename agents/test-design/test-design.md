# test-design — テスト設計スキル (ドキュメント作成)

## サブエージェント実行前提

このスキルは原則 `dev-workflow` オーケストレータから **別エージェント (サブエージェント) として spawn される** ことを想定する。

重要:
- コンテキストはフレッシュ。必要情報はブリーフとファイルから取得すること。
- スコープは原則 **1機能 (`<FID>`) ずつ**。
- 状態は必ず `.dev-workflow/features/<FID>/status.json` に書き戻す。
- 作業終了時は以下を返す: `summary` / `updated_files` / `open_questions` / `next_action` / `blockers`。
- 重要度 high の不明点は即時 ユーザに確認 (チャットで質問)、軽微なものは `open-questions.md` に追記。

## 役割
詳細設計をもとに、機能単位で以下の3層の **テスト設計ドキュメント** (テストケース一覧 + 観点 + 期待結果) を作成する。
**実行可能なテストコードは書かない**。コードへの落とし込みは次フェーズの `test-implementation` が担当する (characterization モードでは `conformance-test`)。本フェーズの責務はあくまで「何をテストすべきかを言語化して合意すること」。

## 実行モード (forward / characterization)

ブリーフの `mode` で挙動が分かれる。**指定がなければ `forward`**。

| | `mode: forward` (デフォルト・TDD) | `mode: characterization` (reverse-design-workflow 用) |
|---|---|---|
| インプット | forward で作成された設計ドキュメント | `reverse-design` が逆生成/修正した設計ドキュメント (ブリーフで指定) |
| 期待値の源泉 | 設計 + 要件 (これから作る正解) | **設計書の主張をそのまま転記** (既存コードの記述) |
| 期待状態 | 後続 test-implementation で **Red** (実装が無いので Fail) | **PASS** (設計が実コードと一致していれば通る) |
| 対象層 | 3 層すべて (unit / integration / e2e) | **ブリーフの `layer` で指定された 1 層のみ** (detailed→unit / basic→integration / requirements→e2e) |
| コード化の担当 | `test-implementation` | `conformance-test` |
| Step 5 (ユーザレビュー) / Step 6 (進捗確定) | 実施する | **スキップ** (承認は reverse-design-workflow の human-checkpoint 側。status.json 更新はブリーフの指示に従い、戻り値で conformance-test の spawn を要請する) |

**characterization モードの規律:**

1. 期待値は **検証対象の設計ドキュメントに書かれた主張のみ** から起こす。**ソースコードを覗いて期待値を決めることは禁止** (それでは設計の検証にならない。conformance-test 側の「カンニング禁止」と同じ理由)
2. 設計書の記述が曖昧で期待入出力を確定できないケースは、期待値を捏造せず `blockers` で `reverse-design` への差し戻しを要請する (conformance-test の `spec_incomplete` 判定に相当)
3. conformance が `mismatch` になった後の再 spawn では、**修正後の設計** に合わせて該当ケースのみ仕様を更新する (修正順序: 設計書 → テスト仕様書 → テストコード、の 2 段目を担う)
4. 出力先・ID 規約・記述項目 (§「成果物」「ID 規約」、Step 3) は forward と共通 (`docs/03_test_design/<FID>/<layer>-test.md`)。conformance-test はこのパスを読む
5. 本文中の TDD (Red 先行) に関する記述は characterization モードには適用しない

## バッチ実行と横断認識

本スキルはフェーズバッチで呼ばれる。複数機能のテスト設計が同時並行で進む前提で:

1. **他機能の同フェーズ成果物 (`docs/03_test_design/F<他>/*.md`) があれば Read** し、テストID採番・観点列挙の粒度・期待結果の書き方を揃える
2. **共通テスト観点・共通フィクスチャ・共通アサーション** に気付いたら `open-questions.md` に `[COMMON 候補]` で追記
3. 既存 `COMMON` 機能のテスト設計 (`docs/03_test_design/COMMON/`) があれば、自機能ではそれを再利用する形にする (重複ケースを書かない)

| 層         | **インプット (検証対象の設計層)**       | 対象                                              | 観点                                                       |
| ---------- | -------------------------------------- | ------------------------------------------------- | ---------------------------------------------------------- |
| 単体テスト | **詳細設計** (機能内部の振る舞い)       | 関数/クラス/モジュール、ガード条件、例外処理       | **詳細設計通りに動くこと**。ロジック網羅、境界値、異常系、状態遷移、UI バリデーション、DB スキーマ制約 |
| 結合テスト | **基本設計** (機能間連携・アーキ要件)   | 機能間 I/F、外部システム連携、アーキ層境界、データフロー | **基本設計通りに繋がること**。機能間整合性、データ永続化、トランザクション境界、副作用、コンポーネント連携 |
| E2E テスト | **要件 (USDM / ユースケース)**          | エンドユーザシナリオ (UI / 公開 API 起点)         | **要件を満たすこと**。ユースケース達成、業務シナリオ、UX、受入条件 |

> **3 層と 3 設計層の対応原則:**
> - 単体テスト = **詳細設計** を検証 (機能内部の振る舞いが詳細設計どおりか)
> - 結合テスト = **基本設計** を検証 (機能間連携・アーキが基本設計どおりか)
> - E2E テスト = **要件** を検証 (システムがユーザ要件を満たすか)
> このピラミッド対応により、設計の各層が必ず対応するテスト層でカバーされる。

## 手順

### Step 1 : インプットの読み込み (3 層別)

3 層それぞれが検証すべき設計層が異なるため、**3 層分すべての設計ドキュメントを読み込む**。

**単体テスト用 (詳細設計)** — 対象機能 `<FID>` の以下を Read:
- `docs/02_detailed_design/<FID>/detailed-design.md` (9章構成: §3-5 サブ機能・例外/エラー処理、§6 I/F 定義、§7-8 シーケンス — 単体側は呼び出し詳細を見る、§9 状態遷移)
- `docs/02_detailed_design/<FID>/ui-design.md` (UI バリデーション、UI あり時)
- `docs/02_detailed_design/<FID>/db-design.md` (DB スキーマ制約、DB あり時)

**結合テスト用 (基本設計)** — プロジェクト全体の以下を Read:
- `docs/01_basic_design/system-overview.md` (システム全体方針)
- `docs/01_basic_design/feature-list.md` (機能一覧と **機能間依存**)
- `docs/01_basic_design/system-architecture.md` (アーキ図、コンポーネント境界、I/F、データフロー)
- `docs/01_basic_design/non-functional.md` (性能/セキュリティ要件)

**E2E テスト用 (要件)** — 以下を Read:
- `docs/requirements/requirements.md` (またはユーザ指定の要件定義書ファイル)
- USDM 形式の場合: 要件 ID `R-###` と仕様 ID `S-###-##` の一覧

これらが揃わない場合、欠けているものを `open-questions.md` に追記してユーザに確認 (E2E テストは要件無しに書けない)。

### Step 2 : テストケース抽出

#### 単体テストの観点 — **詳細設計** を網羅検証

> **検証対象 = 詳細設計**。`detailed-design.md` の各要素 (サブ機能 / 例外 / I/F / 状態遷移) と任意ドキュメント (UI バリデーション / DB 制約) を漏れなくテストに落とす。

- サブ機能 (§5) ごとに最低: 正常系1 + 境界値1 + 異常系1
- **例外/エラー処理表** (§5) の各エラーID `E00x` に対して1テスト以上
- インタフェース定義 (§6) の各 I/F に対して入出力仕様どおりかのテスト1以上
- 状態遷移 (§9) の各遷移 (ガード条件含む) に対して1テスト以上
- 純粋な計算ロジックは入力境界値を網羅
- UI バリデーション規則 (`ui-design.md`) があれば各規則に対し1テスト
- DB スキーマ制約 (NOT NULL / UNIQUE / FK / CHECK) の違反パスを Repository 層で検証

#### 結合テストの観点 — **基本設計** を網羅検証

> **検証対象 = 基本設計**。基本設計のシステムアーキテクチャ・機能間連携・データフローを漏れなくテストに落とす。
> 単体では検出できない「機能間で噛み合っているか」を確認する層。

- システムアーキテクチャ (`system-architecture.md`) の **各 I/F** (機能↔機能、機能↔外部) ごとに1テスト以上
- アーキ図の **各コンポーネント境界** (API ↔ Service / Service ↔ Repository / Service ↔ 外部 SaaS など) のシナリオを最低1テスト
- 機能一覧 (`feature-list.md`) の **機能間依存** (例: F002 が F001 の出力を入力とする) が表現されている箇所は、連携シナリオを結合テストで確認
- シーケンス図の各UC (正常+異常) を結合テストとして起こす
- DB永続化が絡む場合、永続化前後の状態を検証
- 外部システム連携がある場合、連携シナリオごとに1テスト
- トランザクション境界、ロールバック挙動
- 非機能要件 (`non-functional.md`) の性能/セキュリティが結合層で検証可能なものはここで (DB 索引による応答時間、認可境界 など)

#### E2E テストの観点 — **要件** を網羅検証

> **検証対象 = 要件定義書**。すべての要件が **最低 1 シナリオ** でカバーされ、システムがユーザ要件を満たすことを確認する。

- 要件定義書のユースケース or ユーザストーリーごとに1シナリオ
- **USDM の場合**: 要件 ID `R-###` ごとに最低1 E2E シナリオを起こす (1 R に複数 S があれば S-###-## の主要分岐をカバー)
- UI/APIの操作ステップを「人が再現できる」レベルで書く
- 業務シナリオ (複数機能をまたぐもの) はここに入れる
- 受入条件 (Given-When-Then の When/Then が明確なもの) は E2E テストの期待結果に転記

### Step 3 : 各ドキュメント作成

テンプレートをコピーし、各テストケースを以下の項目で記述:

- ID
- 対象 / シナリオ
- 観点 (正常 / 境界値 / 異常 / 連携 / 性能 / セキュリティ)
- 前提条件
- 入力 / 操作手順
- 期待結果
- 関連設計 (どの設計ドキュメントの何節に紐づくか)
- カバーする要件ID (トレーサビリティ)

### Step 4 : トレーサビリティ確認 (3 層別)

各ドキュメント末尾のトレーサビリティ表を埋める。**3 層それぞれに対応する設計層との双方向トレーサビリティ** を確認:

#### 単体テスト ↔ 詳細設計
- 詳細設計のサブ機能ID / 例外ID `E00x` / 状態遷移 / UI バリデーション規則 / DB 制約 → どの UT-<FID>-NNN でカバーされているか
- UT-<FID>-NNN → どの詳細設計要素に対応するか

#### 結合テスト ↔ 基本設計
- 基本設計の I/F / アーキ層境界 / 機能間依存 → どの IT-<FID>-NNN でカバーされているか
- IT-<FID>-NNN → どの基本設計要素 (アーキ図のどのエッジ / どの機能間連携) に対応するか

#### E2E テスト ↔ 要件
- 要件ID (USDM `R-###` / ユースケースID) → どの E2E-<FID>-NNN でカバーされているか
- E2E-<FID>-NNN → どの要件 / 受入条件に対応するか

各層でカバーされていない要素があれば、テストを追加するか、`open-questions.md` で「この要素はどの層でテストすべきか」をユーザに確認する。
特に **E2E テストの要件カバレッジ 100%** は必須 (1 要件 = 0 テスト は禁止)。

### Step 5 : ユーザレビュー (チェックポイント確認)

3層揃った段階でユーザに提示。特に:
- カバレッジ目標 (単体テストの分岐網羅% など) はユーザに必ず確認
- 性能・セキュリティテストの実施有無
- 自動化の方針 (どこまで自動、どこから手動)

決定事項は `decisions.md` に追記。

### Step 6 : 進捗確定 (本フェーズ作業の完了)

1. `status.json` を更新:
   - `phases.test_design.subtasks.{unit_test, integration_test, e2e_test}` を `completed`
   - `phases.test_design.status = "completed"`
   - **`current_phase` はまだ `test_implementation` に進めない** (test-design-review の pass を待つ)
2. `project.json` の `updated_at` 更新。
3. 戻り値で「test-design-review を spawn してほしい」とオーケストレータに伝える。

**重要**: 次フェーズ (`test_implementation`) に進めるのは **`test-design-review` の pass を確認した後** だけ。
