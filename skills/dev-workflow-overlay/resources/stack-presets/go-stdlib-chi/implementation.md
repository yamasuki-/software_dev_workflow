# implementation — Go + chi rules

## ADD
- 全 public 関数の第一引数で `ctx context.Context` を受ける (DB / HTTP / 外部 I/O を伴うもの全て)
- `error` の戻りを `_` で捨てない (errcheck 必須)
- error の包み方:
  - 上位に伝える場合: `return fmt.Errorf("create user: %w", err)`
  - sentinel error: `var ErrUserNotFound = errors.New("user not found")` を `internal/domain/` で定義
  - 判定: `errors.Is(err, ErrUserNotFound)` / `errors.As(err, &target)`
- DB アクセスは sqlc 生成コードを Repository ラッパ経由で使う
  - 直接 `db.Query` / `db.Exec` を書かない (sqlc に集約)
  - Repository ラッパは Service 定義のインタフェースを満たす
- HTTP ハンドラ:
  - request パース → validate → service 呼び出し → response 変換 の順
  - ビジネスロジックを handler に書かない (Service へ)
  - エラーは middleware で domain error → HTTP status に変換 (handler で status 直書きしない)
- ログ:
  - `slog.Default()` または DI された logger を使う
  - request_id を ctx 経由で必ず付与 (`logger.With("request_id", reqID)`)
- 並行処理:
  - goroutine 起動時は必ず ctx を渡し、キャンセル伝搬させる
  - errgroup を使うときは `errgroup.WithContext` で派生 ctx を使う
- グローバル変数禁止:
  - 設定 / DB / Logger は main で組み立てて DI で配る
- panic:
  - リクエスト処理中に panic しない
  - middleware で recover し 500 を返す (recover 漏れ防止)

## OVERRIDE
- ベース「コード生成時に最小実装で Green を目指す」→ Go では Red 確認段階で型シグネチャ (struct, interface, 関数定義) が固まっている前提

- なし

## ADDITIONAL_ARTIFACTS

## REVIEW_EXTRAS
- `golangci-lint run` が 0 件
- `gofmt -l .` / `goimports -l .` が 0 件
- `go vet ./...` が 0 件
- `go test -race ./...` が pass
- error の `_` 捨てがない (errcheck)
- ctx が全層で第一引数として渡っているか
- goroutine が ctx キャンセルで終了するか (リーク検証)
- グローバル変数を新規導入していないか
- secrets が環境変数経由か (`os.Getenv` を main で読んで config 構造体に詰めるパターン)
- panic がリクエスト処理中に出ない設計か (middleware recover が効くか)
- sqlc 生成物が手動編集されていないか (`// Code generated by sqlc. DO NOT EDIT.` の維持)
