# test-design — テスト設計スキル (ドキュメント作成)

## サブエージェント実行前提

このスキルは原則 `dev-workflow` オーケストレータから **別エージェント (サブエージェント) として spawn される** ことを想定する。

重要:
- コンテキストはフレッシュ。必要情報はブリーフとファイルから取得すること。
- スコープは原則 **1機能 (`<FID>`) ずつ**。
- 状態は必ず `.dev-workflow/features/<FID>/status.json` に書き戻す。
- 作業終了時は以下を返す: `summary` / `updated_files` / `open_questions` / `next_action` / `blockers`。
- 重要度 high の不明点は即時 ユーザに確認 (チャットで質問)、軽微なものは `open-questions.md` に追記。

## 役割
詳細設計をもとに、機能単位で以下の3層の **テスト設計ドキュメント** (テストケース一覧 + 観点 + 期待結果) を作成する。
**実行可能なテストコードは書かない**。コードへの落とし込みは次フェーズの `test-implementation` が担当する。本フェーズの責務はあくまで「何をテストすべきかを言語化して合意すること」。

| 層         | 対象                                     | 観点                              |
| ---------- | ---------------------------------------- | --------------------------------- |
| 単体テスト | 関数/クラス/モジュール                   | ロジック網羅、境界値、異常系      |
| 結合テスト | 機能内の複数コンポーネント / 外部I/O     | 連携シナリオ、データ永続化、副作用 |
| E2E テスト | エンドユーザシナリオ (UI / 公開API起点) | ユースケース達成、UX、業務シナリオ |

## 成果物 (`docs/03_test_design/<FID>/`)

| ------------------------ | ---------------------------------------------- |
| `unit-test.md`           | `templates/test-design/unit-test.md`           |

## 手順

### Step 1 : 詳細設計の読み込み

対象機能 `<FID>` の以下を Read:
- `docs/02_detailed_design/<FID>/functional-design.md`
- `docs/02_detailed_design/<FID>/state-transition.md`
- `docs/02_detailed_design/<FID>/sequence.md`
- `docs/02_detailed_design/<FID>/ui-design.md` (UIあり)
- `docs/02_detailed_design/<FID>/db-design.md` (DBあり)
- `docs/01_basic_design/non-functional.md` (性能/セキュリティ要件)

### Step 2 : テストケース抽出

#### 単体テストの観点
- 機能設計のサブ機能ごとに最低: 正常系1 + 境界値1 + 異常系1
- 機能設計の **例外/エラー処理表** の各エラーID `E00x` に対して1テスト以上
- 状態遷移の各遷移 (ガード条件含む) に対して1テスト以上
- 純粋な計算ロジックは入力境界値を網羅

#### 結合テストの観点
- シーケンス図の各UC (正常+異常) を結合テストとして起こす
- DB永続化が絡む場合、永続化前後の状態を検証
- 外部システム連携がある場合、連携シナリオごとに1テスト
- トランザクション境界、ロールバック挙動

#### E2E テストの観点
- 要件定義書のユースケース or ユーザストーリーごとに1シナリオ
- UI/APIの操作ステップを「人が再現できる」レベルで書く
- 業務シナリオ (複数機能をまたぐもの) はここに入れる

### Step 3 : 各ドキュメント作成

テンプレートをコピーし、各テストケースを以下の項目で記述:

- ID
- 対象 / シナリオ
- 観点 (正常 / 境界値 / 異常 / 連携 / 性能 / セキュリティ)
- 前提条件
- 入力 / 操作手順
- 期待結果
- 関連設計 (どの設計ドキュメントの何節に紐づくか)
- カバーする要件ID (トレーサビリティ)

### Step 4 : トレーサビリティ確認

各ドキュメント末尾のトレーサビリティ表を埋める。以下の双方向を確認:

- 要件ID → どのテストでカバーされているか (取りこぼし防止)
- テストID → どの要件/設計箇所に対応するか (無駄テスト防止)

要件ID単位でカバーされていないものがあれば、テストを追加するか、`open-questions.md` で「この要件はどの層でテストすべきか」をユーザに確認する。

### Step 5 : ユーザレビュー (チェックポイント確認)

3層揃った段階でユーザに提示。特に:
- カバレッジ目標 (単体テストの分岐網羅% など) はユーザに必ず確認
- 性能・セキュリティテストの実施有無
- 自動化の方針 (どこまで自動、どこから手動)

決定事項は `decisions.md` に追記。

### Step 6 : 進捗確定 (本フェーズ作業の完了)

1. `status.json` を更新:
   - `phases.test_design.subtasks.{unit_test, integration_test, e2e_test}` を `completed`
   - `phases.test_design.status = "completed"`
   - **`current_phase` はまだ `test_implementation` に進めない** (test-design-review の pass を待つ)
2. `project.json` の `updated_at` 更新。
3. 戻り値で「test-design-review を spawn してほしい」とオーケストレータに伝える。

**重要**: 次フェーズ (`test_implementation`) に進めるのは **`test-design-review` の pass を確認した後** だけ。

## チェックリスト

- [ ] 機能設計の例外ID `E*` すべてが単体テストで網羅
- [ ] シーケンス図の UC すべてが結合テストまたはE2Eで網羅
- [ ] 要件IDのトレーサビリティ表が双方向で完成
- [ ] カバレッジ目標がユーザ確認済み
- [ ] `decisions.md` に追記済み
- [ ] `status.json` 更新済み
