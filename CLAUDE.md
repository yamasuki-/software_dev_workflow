# 開発ワークフローのルーティング規則

機能変更・不具合修正の依頼を受けたら、まず `feature-add-workflow` / `bugfix-workflow` / `dev-workflow` スキルへの該当を確認する (照合を省略して直接実行に入らない)。

設計書の復元・補完系の依頼 (「設計書がないコードから設計を起こしたい」「既存コードの要件を復元したい」「リバースエンジニアリングしたい」等) は `reverse-design-workflow` に振り分ける。

## overlay の優先

プロジェクトに `.dev-workflow/rules/` (stack 層 / project 層のプロジェクト固有ルール) が存在する場合は、`dev-workflow` の代わりに `dev-workflow-overlay` を起動する (ルールが無ければ素の `dev-workflow` でよい)。

## 自明な修正の例外

変更が **1〜数行** で、かつ **ユーザが解決策を指定済み** の場合はワークフローを起動せず直接修正してよい。ただし以下の 3 点をセットで必ず行う:

1. コード修正
2. 関連設計書の整合更新 (docs/design/ 等プロジェクトの設計ドキュメント。コード行番号の参照があればそのずれも反映)
3. テスト追加・更新と Green 確認

## 例外に該当しない場合

以下のいずれかに当てはまる場合は該当ワークフローを起動する:

- 影響範囲が不明
- 複数機能 (F###) にまたがる
- 解決策が未定 (解析・提案が必要)
- 不具合で再現条件の調査が必要

迷う場合は「ワークフローを通すか直接修正か」をユーザに確認してから着手する。
